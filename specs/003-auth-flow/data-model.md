**Date**: 2026-04-11

## Entities

### ApplicationUser

Extends `IdentityUser` from ASP.NET Core Identity. Stored in the `AspNetUsers` table (managed by Identity migrations).

| Field | Type | Source | Notes |
|-------|------|--------|-------|
| Id | string (GUID) | IdentityUser | Primary key; used as `sub` claim in JWT |
| Email | string | IdentityUser | Unique; used as login identifier |
| NormalizedEmail | string | IdentityUser | Uppercase email for lookups (managed by Identity) |
| EmailConfirmed | bool | IdentityUser | Default `false`; email verification is out of scope |
| PasswordHash | string | IdentityUser | PBKDF2-HMAC-SHA256 hash; managed by `UserManager` |
| SecurityStamp | string | IdentityUser | Invalidates tokens on password change; managed by Identity |
| ConcurrencyStamp | string | IdentityUser | Optimistic concurrency; managed by Identity |
| UserName | string | IdentityUser | Set to same value as Email on creation |
| NormalizedUserName | string | IdentityUser | Uppercase username; managed by Identity |

**Validation rules** (enforced by Identity + `UserManager`):
- Email: required, valid format, unique
- Password: minimum 8 characters (default Identity policy); at least one uppercase, lowercase, digit (configurable)
- Duplicate email → `IdentityResult` error `DuplicateEmail`

**Custom extension fields**: None in v1. Future: `DisplayName`, `TimeZone`, etc.

## Identity Tables Created by Migration (M005)

ASP.NET Core Identity's `IdentityDbContext<ApplicationUser>` migration creates:

| Table | Purpose |
|-------|---------|
| `AspNetUsers` | User accounts (ApplicationUser rows) |
| `AspNetRoles` | Roles (empty for now — no role-based auth in v1) |
| `AspNetUserRoles` | User-to-role junction |
| `AspNetUserClaims` | Per-user custom claims |
| `AspNetUserLogins` | External login providers (not used in v1) |
| `AspNetUserTokens` | Identity framework tokens (not JWTs — internal use) |
| `AspNetRoleClaims` | Role-level claims |

## Value Objects / DTOs (Application Layer)

### LoginRequest

```
email: string (required, email format)
password: string (required, non-empty)
```

### RegisterRequest

```
email: string (required, email format)
password: string (required, min 8 chars)
```

### AuthResponse

```
token: string       — HS256 JWT, Bearer token
expiresAt: datetime — UTC ISO 8601 expiry timestamp
userId: string      — GUID (ApplicationUser.Id)
```

## JWT Claim Structure

Tokens generated by `JwtTokenService` include:

| Claim | Value | Notes |
|-------|-------|-------|
| `sub` | ApplicationUser.Id (GUID string) | User identity; used by all controllers |
| `email` | ApplicationUser.Email | Informational; future display use |
| `exp` | Unix timestamp | Expiry; validated by JwtAuthenticationMiddleware |
| `iat` | Unix timestamp | Issued-at; informational |

## State Transitions

### User Session State (frontend)

```
[No token in localStorage]
  → user visits protected route → redirect to /login
  → user logs in successfully → token stored → redirect to accounts

[Token in localStorage]
  → app loads → token parsed → if not expired → authenticated
  → if expired → token cleared → redirect to /login
  → user clicks logout → token cleared → redirect to /login
  → API returns 401 mid-session → token cleared → redirect to /login
```

## Relationship to Existing Domain

### BankAccount.UserId

The `BankAccount` entity has a `Guid UserId` field that currently comes from a `[FromQuery]` parameter and is stored as-is (no FK constraint to a users table). After this feature:

- `BankAccount.UserId` will be populated from `ApplicationUser.Id` (via JWT claim `sub`)
- The field type remains `Guid` — `ApplicationUser.Id` (inherited from `IdentityUser`) is a string GUID but will be parsed to `Guid` in the controller
- No FK constraint added (cross-module coupling would violate Principle I) — the Auth module is the authority on user identity; BankSync stores the userId as an opaque identifier
