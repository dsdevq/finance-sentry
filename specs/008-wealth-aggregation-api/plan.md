# Implementation Plan: Financial Aggregation and Wealth Overview API

**Branch**: `008-wealth-aggregation-api` | **Date**: 2026-04-20 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/008-wealth-aggregation-api/spec.md`

## Summary

Add two new read-only endpoints to the backend — `GET /api/v1/wealth/summary` (net worth breakdown by provider category) and `GET /api/v1/wealth/transactions/summary` (expense/income totals for a date window) — both supporting optional filtering by provider category or specific provider name. Feature is pure read-layer: no new database tables, no migrations. Operates on existing `BankAccount` and `Transaction` data. Introduces `WealthController`, `IWealthAggregationService`, `ProviderCategoryMapper`, and `CurrencyConverter` within the existing `FinanceSentry.Modules.BankSync` module.

## Technical Context

**Language/Version**: C# 13 / .NET 9 (backend only — no frontend changes)  
**Primary Dependencies**: ASP.NET Core 9, EF Core 9, MediatR (existing — no new NuGet packages)  
**Storage**: PostgreSQL 14 — read-only queries against existing `BankAccounts` and `Transactions` tables; no new columns or migrations  
**Testing**: xUnit (backend unit + contract tests)  
**Target Platform**: Linux server (Docker), existing Docker Compose stack  
**Project Type**: Web service extension (modular monolith)  
**Performance Goals**: < 500ms p95 for summary endpoints with up to 20 accounts and 6 months of transactions  
**Constraints**: No new NuGet packages; no DB migrations; `BankAccount.Provider` column must exist (added by feature 007 T003) before full provider filtering works  
**Scale/Scope**: Single developer; personal finance app; ~1–20 accounts per user; ~12 months of transactions
